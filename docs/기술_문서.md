# 슈퍼로봇대전 GC 한글패치 기술 문서

## 1. 목표와 기준 이미지

이 패치는 일본판 게임 ID `GRWJD9`의 수정되지 않은 디스크 이미지를 기준으로 작성되었습니다. 영문판 데이터나 영문판 패치를 기반으로 문자열 순서를 추정하지 않고, 일본판의 레코드 구조·포인터·이벤트 순서를 보존한 채 한국어 데이터를 삽입합니다.

원본 판별값은 다음과 같습니다.

- 크기: `1,459,978,240` 바이트
- SHA-256: `AD4CB99FFB3C0383802A2AB87963F98BA417DFC5184ED3FE3DFE077DA02DB229`

## 2. 처리 흐름

```text
일본판 원본 이미지
  -> 대상 파일 추출 및 구조 검사
  -> 한국어 글꼴·런타임 코드북 구성
  -> 대사/메뉴 레코드별 문자열 치환
  -> 포인터와 고정 토폴로지 재구성
  -> UI 이미지의 SCR별 한국어 렌더링 및 타일 재패킹
  -> 로딩 SPR의 분할 C4 이미지 한국어화
  -> Start.dol의 문자열 풀과 이름 선택 로직 패치
  -> 타이틀·배너의 영문판 흔적 제거
  -> 파일별 독립 감사 및 ISO 재조립
  -> 원본 대비 xdelta3 패치 생성
```

저장소의 도구는 위 작업 중 구조 검사, 치환, 재배치, 재조립 및 감사를 담당합니다. 저작권이 있는 원본 파일과 추출 데이터는 제공하지 않습니다.

## 3. 일본판 고유 구조 보존

대사 데이터는 단순한 문자열 목록이 아닙니다. 화자, 표정, 메시지 창 제어, 분기 및 다음 레코드와의 연결 정보가 함께 들어 있습니다. 번역문을 다른 지역판의 순서대로 덮어쓰면 화자와 대사가 섞일 수 있으므로 다음 원칙을 적용했습니다.

- 일본판 레코드 인덱스와 이벤트 순서를 기준으로 매핑
- 각 컨테이너의 정렬 단위와 종료 마커 보존
- 변경 후 레코드 수, 오프셋 범위 및 포인터 도착점 재검사
- 전투 대사처럼 공용 레코드가 사용되는 항목은 특정 주인공 이름에 종속되지 않도록 교정

## 4. 한국어 글꼴과 코드북

게임의 문자열 렌더러는 일반적인 UTF-8/UTF-16 텍스트를 직접 처리하지 않습니다. 패치 도구는 게임이 해석하는 문자 코드와 새 글리프의 위치를 일치시키는 런타임 코드북을 만들고, 한국어 문자열을 그 코드북으로 인코딩합니다.

특히 코드 경계에서 제어값으로 잘못 해석될 수 있는 조합을 피하거나 교정하고, 실제 문자열에 필요한 완성형 한글·숫자·기호가 글꼴에 존재하는지 검사합니다. 감사 단계에서는 인코딩 불가 문자, 잘못된 종료 코드 및 글리프 누락을 실패로 처리합니다.

## 5. 실행 파일과 주인공 이름

주인공 기본 이름과 이름 입력 화면은 실행 파일의 여러 경로에서 참조됩니다. 한 곳의 문자열만 교체하면 성별 선택, 초기화 또는 이름 비교 과정에서 이전 이름이 다시 나타날 수 있으므로 관련 참조를 함께 패치했습니다.

- 남성 선택: `아카츠키 아키미`
- 여성 선택: `아카츠키 아케미`

성별 상태 비트를 확인하는 분기와 별도 문자열 풀을 사용해 두 이름을 구분하고, 기본값 초기화·이름 입력·비교 루틴이 같은 결과를 사용하도록 맞췄습니다. 코드 변경은 승인된 훅과 코드/데이터 공간에 한정하여 감사합니다.

## 6. 영문판 흔적 제거

최종 데이터는 일본판을 기반으로 재구성합니다. 과거 영문판 패치에서 유입될 수 있는 영문 타이틀, 배너 메타데이터, 크레디트 문구 및 알려진 영문 문자열 표식을 검색하고 제거했습니다. 게임 ID와 일본판 파일 배치를 유지하면서 필요한 한국어 자산만 삽입합니다.

메뉴 이미지도 일본판·영문판·현행판을 SCR 단위로 대조했습니다. 각 표시
이미지는 독립 렌더링하지만 기존 아틀라스를 압축하거나 원본 타일 번호를
바꾸지 않습니다. 게임이 정적 SCR 없이 직접 선택하는 전투·동적 메뉴
타일이 있기 때문입니다. 기존 타일을 모두 같은 번호에 둔 채 한국어 타일을
뒤에 추가하고, 번역 대상 SCR만 새 번호로 재매핑합니다. 대형 아틀라스의
14비트 인덱스와 소형 아틀라스의 10비트 인덱스를 구분해 처리하며, 자세한
절차는 [UI 이미지 한국어화](UI_이미지_패치.md)에 정리했습니다.

`Now Loading...`은 한 장의 312×40 이미지처럼 보이지만 파일 안에서는
가로 `256/32/16/8`, 세로 `32/8` 크기의 C4 텍스처 8개로 분할됩니다.
전체 논리 이미지를 `불러오는 중...`으로 렌더링한 뒤 같은 경계로 다시
나누며, SPR 헤더·팔레트·CEL은 바이트 단위로 보존합니다.

## 7. 자동 감사

최종 빌드 전후에 다음 항목을 검사합니다.

- 입력 원본과 주요 산출물의 크기 및 SHA-256
- 컨테이너별 레코드 수와 정렬 조건
- 문자열 포인터가 허용된 데이터 영역을 가리키는지 여부
- 코드북으로 왕복 가능한 문자열과 글리프 존재 여부
- 주인공 이름의 성별 분기 및 이전 이름 잔존 여부
- 알려진 영문판 패치 표식 잔존 여부
- UI·타이틀 메뉴 SCR 484개와 로딩 논리 이미지 1개의 픽셀 왕복 및 팔레트 보존 여부
- 원본 아틀라스 타일 11,872개와 정적 SCR 미참조 비공백 타일 1,383개의 인덱스·바이트 보존 여부
- 비문자 보존 SCR 32개의 바이트 동일 여부
- ISO 파일 시스템에 삽입된 데이터가 빌드 산출물과 동일한지 여부

감사 도구는 에뮬레이터의 화면 확인을 대신하는 것이 아니라, 재현 가능한 바이너리 수준의 오류를 조기에 차단하기 위한 장치입니다.

## 8. xdelta3 릴리스

릴리스 파일은 수정되지 않은 일본판 ISO와 최종 한국어 ISO의 차이를 xdelta3로 인코딩한 것입니다. 로컬 경로가 패치 헤더에 들어가지 않도록 애플리케이션 헤더를 사용하지 않으며, 배포 전 패치를 다시 디코딩해 최종 ISO와 SHA-256이 일치하는지 확인합니다.

적용 스크립트도 같은 방식으로 원본 해시를 먼저 확인하고, 디코딩이 성공한 뒤 결과 해시를 검사합니다. 패치 파일이나 출력 ISO를 Git 저장소에 커밋하지 않습니다.

## 9. 저장소에 포함하지 않는 항목

- 원본 또는 패치된 ISO/GCM 이미지
- 게임에서 추출한 실행 파일, 컨테이너, 이미지 및 글꼴
- 일본어 원문 전체 덤프와 전체 대사 데이터
- xdelta3 실행 파일

이 제한은 원저작물과 제3자 프로그램의 권리를 존중하고, 저장소가 패치 제작 도구와 기술 문서에 집중하도록 하기 위한 것입니다.
